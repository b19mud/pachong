浏览器是如何逐步解码的？
一个例子：<img src=# onerror="eval(<?php echo $_GET['x']; ?>)">
获取的X参数首先是在URL中被接受，然后到一个javascript中，然后再是在一个HTML标签里，那他实际的解码过程会是怎样呢？
第一步，先获取x参数，例如我们接受到的是 %26%2397%3B%26%23108%3Bert('\x68\x65\x6c\x6c\x6f')
可能有人会糊涂了，这到底是个什么屌东西，慢慢来：
浏览器首先对你输入的那串参数 %26%2397%3B%26%23108%3Bert('\x68\x65\x6c\x6c\x6f') 进行一次URL解码，然后就变成了：
&#97;&#108;ert('\x68\x65\x6c\x6c\x6f')
第二步，由于输入的东西是在一个HTML标签里的javascript中，解码时候，HTML标签先是老大，所以浏览器再进行一次HTML解码：
alert('\x68\x65\x6c\x6c\x6f')
这时候，只剩下javascript这个小弟了，浏览器进行JS解码:
alert('hello')
最后渲染出来就是：<img src=# onerror="eval(alert('hello'))">
第二个例子：<div id='here'></div><script>document.getElementById('here').innerHTML= "."'".htmlspecialchars($_GET['x'])."'".";</script>
在这里获取的x参数被htmlspecialchars编码了，所以当x为 <img src=#> 时候，经过一轮htmlspecialchars函数的转换，是不会执行里面的HTML代码的，因为已经面目全非了。
如果我们使x为 \x3c\x69\x6d\x67\x20\x73\x72\x63\x3d\x23\x3e (<img src=#>的JS编码)会发生什么？
首先 在这里，x参数在URL中被接受并编码，然后由javascript控制输出到HTML中，那么在这里对于HTML来说JS就是老大，所以浏览器会在PHP的htmlspecialchars函数过后（由于没有HTML特殊字符，不会有什么变化）先JS解码，就成了<img src=#>，然后输出，因为htmlspecialchars在JS解码之前，所以输出<img src=#>就畅通无阻了。

今日在群里看到某些学弟po出来问问题的代码实在是惨不忍睹，不得不吐槽一番。虽然是C语言的初学者，但是我认为即便是初学者也应该注意代码格式已经命名规则之类的东西。
主要是有几点原因，一是如果不及时养成良好的习惯，以后就改不过来了。二是为了代码的可读性以及可维护性，不过目前的十几行代码并没啥可读性而言。我们假设，你编写了一个计算器的程序，大概几百上千行代码，如果所有的变量都是aa，bb，cc，x，y，z这种命名，第二天保准你记不得每个变量是干啥的。另外，糟糕的缩进和花括号也会让你痛不欲生，所以，尽量在刚开始学习写代码的时候养成良好的习惯。
又叫骆驼命名法，指使用大小写来混合命名。
例如userProfile。把每个单词的首字母大写，而第一个单词不大写，像驼峰一样，故得名。
再给几个例子：myStudentCount，uploadFile等。
如果第一个单词大写，则称为大驼峰，否则为小驼峰。例如：MyStudentCount和UploadFile均为大驼峰。
这个比较随意，指用下划线分割每个单词。例如
user_profile，my_student_count，upload_file等。
匈牙利命名法的核心是，在变量名前加上该变量类型的缩写，使其他人看到变量名可以明白该变量是什么类型。例如：
string sUserProfile;
int iMyStudentCount;
File hUploadFile;
接下来是缩进的问题，通常情况下C语言的缩进都是4个字符，也是被大多数人接受的宽度。
如果确定是4个宽度了，那么不要在代码中出现没有对齐的情况了。
多说一句，不管for和if后面有几行代码，请加上花括号，不要吝啬你的花括号。下面给个比较好一定的代码格式，大家自己领悟吧。
仔细看，for和if的花括号是在同一行，而函数main的花括号换行了，这是另一种编码风格，即函数，代码块等表示一个类似于module的时候，花括号换行。而当你表达一个过程的时候，例如for和if这种语句块，则不换行。
两种风格对于C来说，我偏向前者，但是如果代码写多了，你可能会喜欢第二种了。建议初学者还是采用第一种方法，不要轻易尝试第二种。
